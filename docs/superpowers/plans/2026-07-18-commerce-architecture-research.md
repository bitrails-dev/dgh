# Commerce architecture research — Payload 3.85.1 and storefront boundary

Date: 2026-07-18  
Scope: architecture evidence only. This note does not change implementation.

## Decision

Use Payload's native authentication implementation for `Customers`, remove the custom
password/session/JWT implementation, and remove the Astro application-level BFF.

Do **not** make the browser call a central Payload hostname cross-site. The storefront supports
arbitrary tenant domains, so third-party-cookie behavior would be unreliable and would require a
growing static CORS/CSRF origin list. Instead, route a narrow Payload storefront API prefix through
the deployment ingress on every tenant host. This is transport routing, not an application BFF:
there is no Astro handler, token relay, cookie rewriting, tenant selection, pricing, authorization,
or business logic in the proxy.

The selected public boundary is:

```text
browser
  -> https://<tenant-host>/api/store-cms/*
  -> trusted ingress path rewrite
  -> Payload /api/*
```

The proxy must preserve `Host`, set `X-Forwarded-Host` itself, discard any client-supplied forwarded
headers, and only forward the explicitly allowlisted storefront/auth routes. Payload remains the
only application that authenticates, authorizes, validates CSRF origin, prices, reserves, checks
out, and sets cookies.

## 1. Payload auth can support tenant-local email identity

### Finding

Payload's default auth email is globally unique. The installed 3.85.1 source defines the auth
`email` field with `unique: true`
([source](../../../cms/node_modules/payload/dist/auth/baseFields/email.js#L3-L21)).
Using default email login would therefore prevent the same email from being registered in two
tenants.

Payload 3.85.1 has a native, intentional escape hatch: enable username login and set
`allowEmailLogin: false`. In that mode Payload changes the generated email field to
`unique: false`
([source](../../../cms/node_modules/payload/dist/auth/getAuthFields.js#L17-L34)).
The installed defaults make username required and globally unique
([source](../../../cms/node_modules/payload/dist/collections/config/defaults.js#L152-L160)).

### Required configuration

Configure `Customers.auth` exactly as follows:

```ts
auth: {
  loginWithUsername: {
    allowEmailLogin: false,
    requireEmail: true,
    requireUsername: true,
  },
  maxLoginAttempts: 5,
  lockTime: 10 * 60 * 1000,
  tokenExpiration: 7 * 24 * 60 * 60,
  useSessions: true,
  removeTokenFromResponses: true,
  cookies: {
    sameSite: 'Lax',
    secure: true,
  },
}
```

Use the hidden, server-generated username as the global login key:

```text
<tenant-id>:<normalized-email>
```

The username must be generated by a collection hook from the resolved tenant and normalized email;
never accept a browser-supplied username. A tenant rename then cannot invalidate login identity.
The shopper UI continues to ask only for email and password. It derives no security decision from
the email: the tenant is resolved from the trusted request host, and the CMS registration/login
adapter supplies the generated username to Payload.

This keeps tenant-local duplicate emails while retaining Payload's native scrypt password handling,
login-attempt lockout, sessions, JWT signing, cookie generation, refresh, logout, reset-password,
and verification machinery. Payload documents customer accounts as an intended auth-collection use
case and exposes auth operations through REST, Local API, and GraphQL
([Authentication overview](https://payloadcms.com/docs/authentication/overview),
[Authentication operations](https://payloadcms.com/docs/authentication/operations)).

### Small adapter that remains

Two Payload-side storefront endpoints remain justified:

- `POST /api/commerce/store/:tenantSlug/auth/register`
- `POST /api/commerce/store/:tenantSlug/auth/login`

They are not an auth implementation. They resolve the tenant, construct the hidden username, invoke
Payload's native create/login operations, and return Payload's native response/cookie. Registration
must overwrite tenant and username server-side. Login must reject when the host-resolved tenant and
path tenant differ. Do not retain custom password hashing, password verification, signed session
tokens, `X-Session-Token`, dummy hashes, or custom session parsing.

Prefer the built-in endpoints directly where their request shape is sufficient:

- `POST /api/customers/logout`
- `GET /api/customers/me`
- `POST /api/customers/refresh-token`
- `POST /api/customers/forgot-password`
- `POST /api/customers/reset-password`
- `POST /api/customers/verify/:token`

For forgot-password, the UI must submit the generated tenant-qualified username, not plain email,
because email is deliberately non-unique. The browser should call a narrow Payload endpoint that
accepts email, resolves tenant from the trusted host, and invokes Payload forgot-password by
username; it must always return the same success response whether an account exists.

## 2. Multi-tenant plugin compatibility

The plugin does not prohibit additional auth collections. It selects the configured admin user
collection separately
([installed source](../../../cms/node_modules/@payloadcms/plugin-multi-tenant/dist/index.js#L50-L73))
and injects a tenant relationship into every collection listed in its `collections` configuration
([installed source](../../../cms/node_modules/@payloadcms/plugin-multi-tenant/dist/index.js#L133-L176)).
The official plugin documentation likewise describes tenant-field injection and automatic tenant
assignment for configured collections
([Multi-Tenant Plugin](https://payloadcms.com/docs/plugins/multi-tenant)).

Therefore:

- Keep `Users` as `admin.user`; do not make `Customers` the admin user collection.
- Keep `customers: {}` in the multi-tenant plugin collection map.
- Keep customer collection access separate from admin-user tenant membership.
- Enforce customer self-read/self-update using `req.user.collection === 'customers'`,
  `req.user.id`, and the token-bound customer tenant.
- Never depend on the URL tenant alone after authentication.

Required proof test: create the same normalized email in tenant A and tenant B, log into each from
its host, and prove `/me`, update, cart, checkout, order history, forgot-password, and logout cannot
cross tenants.

## 3. Cookie, JWT, session, CORS, and CSRF behavior

Payload's auth cookie is HTTP-only, is set by login/logout/refresh/me operations, and can be included
with browser `fetch(..., { credentials: 'include' })`. Payload explicitly states that HTTP-only
cookies mitigate token theft but do not alone prevent CSRF
([Cookie strategy](https://payloadcms.com/docs/authentication/cookies)).

Installed 3.85.1 supports cookie `domain`, `sameSite`, and `secure`
([types](../../../cms/node_modules/payload/dist/auth/types.d.ts#L173-L179)), sessions, token expiry,
and login lockout
([types](../../../cms/node_modules/payload/dist/auth/types.d.ts#L210-L253)). Its generated cookie is
HTTP-only and path `/`; omitting `domain` makes it host-only
([source](../../../cms/node_modules/payload/dist/auth/cookies.js#L55-L89)).

Payload's `csrf` configuration is an explicit string-array origin allowlist, while `cors` is a
separate allowlist
([types](../../../cms/node_modules/payload/dist/config/types.d.ts#L950-L953)). Cookie
authentication checks `Origin` against that CSRF list; when Origin is absent and CSRF is configured,
it accepts only `Sec-Fetch-Site: same-origin|same-site|none`
([source](../../../cms/node_modules/payload/dist/auth/extractJWT.js#L11-L36)).

Consequences:

- A central Payload origin plus arbitrary unrelated storefront domains is not a robust direct-cookie
  design. Payload's docs require `SameSite=None; Secure` and explicit CORS origins for cross-domain
  cookies, while browsers may still block third-party cookies
  ([Cookie strategy](https://payloadcms.com/docs/authentication/cookies)).
- Same-origin ingress routing avoids third-party cookies and CORS entirely.
- Configure customer cookies as host-only (`domain` omitted), `Secure`, `SameSite=Lax`.
- Keep Payload's CSRF origin check. The ingress must pass the original `Origin`; never replace an
  arbitrary client Origin with the CMS origin.
- Populate `csrf` with the controlled production storefront origins. For arbitrary custom tenant
  domains, deployment automation must regenerate this allowlist from the authoritative tenant-domain
  registry before enabling commerce on that domain. Until that automation exists, commerce on a new
  domain remains disabled.
- Never configure `cors: '*'` with credentials. The same-origin route needs no browser CORS grant.
- Set `removeTokenFromResponses: true`; browser JavaScript receives no reusable JWT.

## 4. Custom endpoint authentication

Custom Payload endpoints are not automatically authorized merely because they live inside Payload.
Handlers receive `req.user`; each protected commerce handler must reject missing users and enforce
customer ID plus tenant ownership. Payload's custom-endpoint documentation states that Payload
automatically supplies the request object and that endpoint code owns its response behavior
([Custom endpoints](https://payloadcms.com/docs/custom-components/custom-endpoints)).

Required endpoint rules:

- Public: catalog reads, quote, registration, login, forgot-password, payment webhooks.
- Customer-authenticated: `/me`, customer cart persistence, checkout association, order history,
  address book, downloadable fulfillment.
- Admin-authenticated and tenant-scoped: commerce operations in the admin/API.
- Payment webhooks: gateway-signature authentication only; never customer-cookie authentication.
- Every authenticated query must include tenant and customer/order ownership constraints.
- Every tenant slug must be compared to the host-resolved tenant before any read or mutation.

## 5. Rate limiting

Payload native auth provides per-account failed-login lockout (`maxLoginAttempts`, default 5 in the
installed source), not generic IP/request rate limiting
([installed defaults](../../../cms/node_modules/payload/dist/collections/config/defaults.js#L122-L142)).
No generic rate-limit option exists in the installed root config type. Therefore Payload does not
replace edge abuse controls.

Required architecture:

- Delete the Astro in-memory `Map` limiter; it is process-local and currently documents that ceiling
  itself ([repo source](../../../src/lib/store/server.ts#L115-L130)).
- Apply shared ingress/CDN limits by normalized client IP:
  - login/register/forgot-password: 10 requests/minute/IP and Payload account lockout;
  - quote/cart mutation: 60/minute/IP;
  - checkout: 10/minute/IP plus one in-flight checkout/order idempotency key;
  - catalog search: 120/minute/IP;
  - webhook: gateway-specific limit high enough for retries; signature validation remains mandatory.
- Return `429` and `Retry-After`.
- Trust `X-Forwarded-For` only from the known ingress; overwrite client-supplied values.

This is a deployment acceptance gate. Do not enable tenant commerce if the shared limiter is absent.

## 6. Payload jobs retry semantics

Payload jobs persist status, errors, and retries. A task is retried only when its handler fails and
the task/workflow has retries configured. Official docs define `retries` and state that successful
handlers complete while failed handlers retry or become failed
([Jobs Queue overview](https://payloadcms.com/docs/jobs-queue/overview),
[Tasks](https://payloadcms.com/docs/jobs-queue/tasks)).

The current CMS already defines a commerce queue runner and `processPaymentEventTask`, but a caught
and suppressed exception is a successful task from Payload's perspective. Payment processing must:

1. make the signed event insert idempotent;
2. execute order-state and inventory side effects transactionally where possible;
3. throw on any incomplete required side effect;
4. configure bounded exponential retries;
5. mark the event fully processed only after side effects commit;
6. retain a terminal failure for reconciliation and alerting;
7. provide a scheduled reconciliation task for captured transactions whose orders/inventory are not
   finalized.

The runner itself is also a deployment gate. Payload documents that queued jobs do nothing until a
runner executes them and recommends a separate process for dedicated servers
([Queues](https://payloadcms.com/docs/jobs-queue/queues)). In-process `autoRun` is acceptable only for
the current single persistent Node deployment; serverless deployment requires an external cron or
dedicated worker.

## 7. Feasibility against the current repo

The Astro application is SSR using the Node standalone adapter
([config](../../../astro.config.mjs#L1-L12)). It resolves tenants from the incoming host and currently
implements an application BFF that owns cart/session/CSRF cookies, relays a custom token, and applies
an in-memory limiter
([source](../../../src/lib/store/server.ts#L1-L130)). The CMS URL is server-only
([environment example](../../../.env.example#L6-L10)).

Removing those Astro API handlers is feasible if and only if deployment provides the same-origin
ingress route and shared rate limiting described above. No deployment/reverse-proxy configuration is
currently present in the repository, so this is an explicit implementation and verification task,
not an assumption.

### No-go fallback

If the production platform cannot route a narrow same-origin prefix to Payload while preserving the
tenant host and enforcing the proxy header contract, keep a **thin transport proxy** in Astro.
It may stream requests/responses and enforce the same shared rate limiter, but it must not own auth,
cookies, session signing, tenant claims, pricing, or commerce logic. This fallback is not permission
to retain the current BFF.

## 8. Acceptance evidence required before deleting the BFF

1. A deployed test tenant proves `Set-Cookie` is host-only, `Secure`, `HttpOnly`, and `SameSite=Lax`.
2. Browser requests use `credentials: 'include'`; no JWT appears in JSON, storage, logs, or client JS.
3. Cross-origin POST with a valid cookie is rejected; same-origin POST succeeds.
4. A malicious `Origin`, `Host`, `X-Forwarded-Host`, and `X-Forwarded-For` cannot select another
   tenant or bypass CSRF/rate limits.
5. Two tenants can register the same email and remain fully isolated.
6. Login lockout and shared IP throttling both work across two application instances.
7. Logout revokes the current Payload session; `allSessions=true` revokes all sessions when exposed.
8. Password reset and email verification links return to the correct tenant host.
9. The allowlisted proxy prefix cannot reach admin, GraphQL, jobs runner, migrations, arbitrary
   collections, or internal commerce collections.
10. The BFF files and custom crypto/password/session code are deleted only after all preceding checks
    pass in the deployed topology.

## Final recommendation

The architecture is feasible and should proceed, with one non-negotiable infrastructure dependency:
same-origin ingress routing plus a shared edge rate limiter. Payload replaces the custom auth stack,
but not generic rate limiting and not deployment routing. Tenant-local duplicate emails are supported
without custom password/session code by using Payload username login with a hidden
tenant-ID-qualified username and non-unique contact email.
